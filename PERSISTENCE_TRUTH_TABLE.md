# PERSISTENCE TRUTH TABLE

> Documents the actual persistence mechanism for each NOMOS subsystem.
> No feature is described as durable unless it is actually durable.

Generated by inspection pass — March 2026.

---

## Summary

**All current persistence in NOMOS is client-side localStorage or in-memory UI state.**

There is no subsystem that writes governance records, audit records, or evaluation history to a server-side database. The `lib/db` (Drizzle + PostgreSQL) exists in the workspace but is not currently wired to any governance or audit feature.

## What the Labels Mean

| Label | Meaning |
|-------|---------|
| `DURABLE` | Persists to a real database (server-side). Survives restarts. |
| `IN_MEMORY_CLIENT_ONLY` | Lives in browser localStorage only. Lost if storage is cleared. Not shared across sessions/devices. |
| `IN_MEMORY_ONLY` | Lives in React UI state only. Lost on page refresh. |
| `DERIVED_AT_RUNTIME` | Computed fresh from persistent inputs on each call. Has no independent storage. |
| `STUB / NOT FULLY WIRED` | Infrastructure exists but is not connected to the feature. |

## Documentation Alignment

The following language in existing docs must be read with this table in mind:

| Claim | Accurate reading given this table |
|-------|----------------------------------|
| "Immutable audit history" | Append-only writes enforced; not tamper-proof across storage clears |
| "Immutable governance audit records" | Append-only within a browser session; client-side only |
| "Session worklog — immutable operational trace" | In-memory only; lost on refresh |
| "Historical records are immutable" (Constitutional Law IV) | Enforced by append-only write pattern; not enforced by durable storage |

## Path to Durable Persistence

The `lib/db` package (Drizzle + PostgreSQL) is available and schema-ready. Wiring the governance audit trail to server-side persistence would require:

1. Adding a `governance_audit_records` table to `lib/db/src/schema.ts`
2. Adding a `POST /api/nomos/audit/record` route to `artifacts/api-server`
3. Replacing `governance_audit_trail.ts` localStorage writes with API calls
4. Doing the same for `audit_store.ts` evaluation records

This is a future task. Current behavior (localStorage) must be documented accurately until then.
