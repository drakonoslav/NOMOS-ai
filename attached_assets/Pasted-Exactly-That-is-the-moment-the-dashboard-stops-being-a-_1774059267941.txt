The correct scenario set

You want exactly these three:

Scenario A — lawful_baseline

Target:
	•	verification.status = "LAWFUL"
	•	authority = AUTHORIZED
	•	action outcome = APPLIED

Use:
	•	low epsilon
	•	full identifiability
	•	high model confidence
	•	robust candidate survives comfortably

⸻

Scenario B — degraded_low_margin

Target:
	•	verification.status = "DEGRADED"
	•	authority = CONSTRAINED
	•	action outcome = DEGRADED_ACTION_APPLIED

Use:
	•	feasibility still true
	•	robustness just above or near threshold
	•	moderate uncertainty
	•	lower model confidence
	•	partial identifiability

This is the most important one.

⸻

Scenario C — refused_infeasible

Target:
	•	verification.status = "INVALID"
	•	authority = REFUSED
	•	action outcome = REFUSED

Use:
	•	hard feasibility failure
	•	no candidate survives screening
	•	or verification sees broken legality at the lowest layer

Where to wire it in the dashboard

Best UX pattern

Put scenario selection in two places:

1. Top bar

Compact selector:
	•	Lawful
	•	Degraded
	•	Invalid

2. Overview page

More descriptive scenario cards:
	•	Lawful Baseline
	•	Degraded Low Margin
	•	Refused Infeasible

That gives you:
	•	quick switching
	•	explanatory selection

⸻

Recommended state shape

Add an app-level scenario type:

export type DemoScenario =
  | "lawful_baseline"
  | "degraded_low_margin"
  | "refused_infeasible";

And a state container:

interface ScenarioRuntimeState {
  scenario: DemoScenario;
  dashboardState: DashboardState;
}

Then build:

function buildScenarioRuntime(scenario: DemoScenario): DashboardState

That function should return:
	•	belief
	•	observer result
	•	model confidence
	•	proposals
	•	decision
	•	verification
	•	authority
	•	audit records

all in a deterministic way.

⸻

Best implementation strategy

Do not make the UI itself invent scenarios.

Instead:

Build a scenario adapter layer

Suggested files:

src/demo/
  scenario_types.ts
  scenario_builder.ts
  lawful_baseline.ts
  degraded_low_margin.ts
  refused_infeasible.ts

Each scenario file returns a complete runtime snapshot.

Then the dashboard just renders the selected snapshot.

That keeps the UI clean and the system architecture honest.
