Build repair-as-graph transformation for NOMOS.

Goal:
Represent repairs as explicit graph transformations instead of text-only suggestions.

This means NOMOS must be able to:
- propose structural graph edits
- connect them to a violated constraint diff
- re-evaluate the modified graph
- determine whether feasibility is restored

Create:
- src/repair/graph_repair_types.ts
- src/repair/graph_repair_planner.ts
- src/repair/graph_repair_executor.ts
- src/repair/graph_repair_validation.ts
- src/tests/graph_repair_test.ts

Requirements:

1. Add canonical repair types.

type GraphRepairActionType =
  | "ADD_ENTITY"
  | "REMOVE_ENTITY"
  | "UPDATE_QUANTITY"
  | "UPDATE_UNIT"
  | "ADD_RELATION"
  | "REMOVE_RELATION"
  | "UPDATE_RELATION_OFFSET"
  | "MOVE_ENTITY_TO_CANDIDATE"
  | "ADD_TAG"
  | "REMOVE_TAG";

interface GraphRepairAction {
  id: string;
  type: GraphRepairActionType;

  targetNodeId?: string | null;
  targetEdgeId?: string | null;

  payload: Record<string, unknown>;
  rationale: string;
}

interface GraphRepairPlan {
  constraintId: string;
  variableName: string;

  actions: GraphRepairAction[];

  expectedRepairEffect: string[];
  estimatedMinimalityScore: number;
}

interface GraphRepairValidationResult {
  repairPlanId: string;
  reEvaluated: boolean;
  restoredFeasibility: boolean;

  remainingViolations: string[];
  summaryLines: string[];
}

2. Build deterministic functions:
- buildGraphRepairPlan(graph, constraintDiff)
- applyGraphRepairPlan(graph, repairPlan)
- validateGraphRepairPlan(graph, repairedGraph, constraintSet)

3. Repair generation rules:

Numeric undershoot:
- if fast carbs are below threshold, prefer:
  - UPDATE_QUANTITY on an existing qualifying entity
  - else ADD_ENTITY with candidate-local ownership and proper relation

Numeric overshoot:
- prefer UPDATE_QUANTITY downward
- else REMOVE_ENTITY if minimal

Window failure:
- prefer UPDATE_RELATION_OFFSET if entity type is correct but outside window

Structural failure:
- restore missing BELONGS_TO_CANDIDATE
- restore moved entity to baseline candidate
- restore missing relation if required

Subset violation:
- REMOVE_ENTITY or REMOVE_RELATION from disallowed item

4. Minimality principle:
Prefer the smallest graph transformation that restores feasibility.

Order:
- change quantity first
- change offset second
- add/remove entity only if needed
- retag only if explicitly allowed by system policy

5. Validation:
After applying repair plan:
- re-run graph-first execution
- record whether the violating constraint is resolved
- record whether any new violations appear

6. Example carb timing repairs:
B:
- reduce oats or replace with fast carb
C:
- move dextrin from -120 min to inside 90-min window
D:
- reduce oats from 30g to <=20g or move oats outside 60-min window

7. Keep everything deterministic.
8. Do not use LLM generation.
9. Repair actions must operate over graph nodes/edges, not just text.

Success condition:
NOMOS can generate graph-native repair plans, apply them structurally, and validate whether they actually restore admissibility.
