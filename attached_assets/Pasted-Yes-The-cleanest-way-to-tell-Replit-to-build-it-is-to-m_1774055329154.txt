For the actual link to a model, the simplest pattern is:
	•	put OPENAI_API_KEY in Replit Secrets,
	•	install the official library,
	•	create a tiny server-side wrapper around client.responses.create(...),
	•	have llm_proposer.ts call that wrapper and convert the returned JSON into NOMOS proposals. Replit’s Secrets are exposed as environment variables in your app, and the OpenAI docs show the Responses API and official JS library as the standard path for Node/TypeScript integrations.  ￼

A minimal server-side wrapper looks like this:

// src/llm/openai_client.ts
import OpenAI from "openai";

const client = new OpenAI({
  apiKey: process.env.OPENAI_API_KEY,
});

export async function generateProposalJSON(input: {
  missionContext: unknown;
  belief: unknown;
  modelSignature: unknown;
}) {
  const response = await client.responses.create({
    model: "gpt-5.4",
    input: [
      {
        role: "system",
        content: `
You are the LLM proposer inside NOMOS.
You may propose plans and hypotheses.
You are not the verifier, decision engine, or guard.
Return only valid JSON.
        `.trim(),
      },
      {
        role: "user",
        content: JSON.stringify(input),
      },
    ],
    text: {
      format: {
        type: "json_schema",
        name: "nomos_proposal_bundle",
        schema: {
          type: "object",
          additionalProperties: false,
          properties: {
            proposals: {
              type: "array",
              items: {
                type: "object",
                additionalProperties: false,
                properties: {
                  kind: { type: "string" },
                  confidence: { type: "string" },
                  rationale: { type: "string" },
                  assumptions: {
                    type: "array",
                    items: { type: "string" }
                  },
                  controlSequence: {
                    type: "array",
                    items: {
                      type: "array",
                      items: { type: "number" }
                    }
                  }
                },
                required: ["kind", "confidence", "rationale", "assumptions"]
              }
            }
          },
          required: ["proposals"]
        }
      }
    }
  });

  return response.output_text;
}

OpenAI’s docs recommend the Responses API for new work, support the official TypeScript/JavaScript SDK, and recommend structured outputs over loose JSON when possible.  ￼

Then in llm_proposer.ts, do not let the model return a CandidatePlan directly. Let it return a proposal bundle, then translate that into your existing proposal objects, and only afterward convert eligible CONTROL_PLAN proposals into CandidatePlan objects for the decision_engine. That preserves your constitutional ordering: model output is suggestion, while screening and authorization remain downstream. The OpenAI docs also note the Responses API supports tool use and structured generation, which fits this proposer-only role well.  ￼

For Replit specifically, tell it to:
	•	add OPENAI_API_KEY in Secrets,
	•	keep the app’s start command in package.json,
	•	and use Replit’s run configuration rather than exposing secret values in code. Replit documents that secrets are available as environment variables, and its app configuration supports explicit run commands.  ￼

A good package.json shape is:

{
  "name": "nomos-core",
  "private": true,
  "type": "module",
  "scripts": {
    "start": "tsx src/main.ts",
    "check": "tsc --noEmit",
    "build": "tsc"
  },
  "dependencies": {
    "openai": "^latest"
  },
  "devDependencies": {
    "tsx": "^4.0.0",
    "typescript": "^5.0.0"
  }
}

That is enough for Replit to run with npm run start, and Replit’s docs confirm package changes can be installed through the run flow or standard package tooling.  ￼
