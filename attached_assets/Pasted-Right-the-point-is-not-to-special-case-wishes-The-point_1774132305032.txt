Right — the point is not to special-case “wishes.”

The point is to build a parser rule that can handle any measurable noun phrase without needing a domain-specific whitelist.

So the design target is:

if a noun can be paired with a quantity and a unit, NOMOS should capture it as a quantified entity

Examples:
	•	30g oats
	•	500 mL water
	•	2 capsules magnesium
	•	40 lb dumbbell
	•	3 hours sleep
	•	12 reps curls
	•	5 mg melatonin
	•	2 bananas
	•	1 container yogurt

And if the input is weird, like 9 grams wishes, the extractor should still preserve it structurally rather than crash or discard it. That is a robustness rule, not the main use case.

The better framing

You do not want:
	•	“food extractor”
	•	“nutrition-only extractor”
	•	“wish parser”

You want:

measurable noun extraction

or

quantified entity extraction

That means:
	•	number
	•	unit
	•	nearby noun phrase
	•	optional modifiers
	•	optional time relation
	•	optional section role

Then later layers decide whether it is:
	•	food
	•	supplement
	•	fluid
	•	time offset
	•	load
	•	repetition
	•	duration
	•	generic measurable object

What to tell Replit instead

Use this revised ask:

Build quantified entity extraction as a core NOMOS parser layer.

Goal:
If text contains a number + unit + nearby measurable noun phrase, NOMOS should extract it as a quantified entity, even if the sentence is messy.

This is not a food-only parser.
This is a general measurable-object parser that must work before domain routing.

Requirements:

1. Add canonical types.

interface QuantifiedEntity {
  id: string;

  rawText: string;
  normalizedText: string;

  amount: number;
  unit: string;
  normalizedUnit: string;

  entityLabel: string;
  normalizedEntityLabel: string;

  category:

    | "supplement"
    | "fluid"
    | "load"
    | "duration"
    | "countable_item"

    | "candidate_item"
    | "constraint_operand"
    | "state_fact"
    | "objective_operand"
    | "unknown";

  modifiers?: string[];
  tags?: string[];

  confidence: "low" | "moderate" | "high";
}

- mg

- oz

- ml, mL

- second, seconds

Training:
- rep, reps

3. Extraction rule:
If a valid quantity + unit pattern is detected, bind the nearest compatible noun phrase as the measurable entity.

Examples that must parse:
- 30g oats
- 80g cyclic dextrin
- 1 banana
- 2 eggs
- 1 container yogurt
- 500 mL water
- 40 lb dumbbell
- 3 hours sleep
- 12 reps curls
- 2 capsules magnesium

4. Robustness rule:
If the phrase is syntactically odd but still clearly quantity-bearing, preserve it rather than dropping it.
Example:
- 9 grams wishes

This is a robustness fallback, not a target domain assumption.

5. The extractor must tolerate:
- messy syntax
- missing punctuation
- unit before/after noun in common patterns
- multiple quantified entities in one sentence

6. Add category inference heuristics:
- food-like nouns -> food
- supplement-like nouns -> supplement
- water / liquid nouns -> fluid
- lb/kg attached to equipment nouns -> load
- hour/minute/day attached to activity/rest/sleep nouns -> duration
- count nouns like eggs / bananas / capsules -> countable_item
- otherwise unknown

7. Add role inference heuristics:
- candidate sentence -> candidate_item
- threshold/constraint section -> constraint_operand
- state section -> state_fact
- objective section -> objective_operand
- else unknown

8. This layer must run before domain-family routing.
Domain routing should use quantified entities as evidence.

9. Add tests for:
- nutrition examples
- supplement examples
- training load examples
- duration examples
- malformed but quantity-bearing examples
- multi-entity candidate strings

Success condition:
Any measurable noun phrase with a quantity and unit can be captured as a quantified entity and made available to later compiler stages.

Why this is better

This version makes the architecture explicit:
	•	it is general
	•	it is unit-driven
	•	it is noun-phrase driven
	•	it is domain-agnostic first
	•	domain comes later

So NOMOS can parse:
	•	physical food items
	•	doses
	•	liquids
	•	loads
	•	durations
	•	repetitions
	•	generic measurable objects

without needing a separate parser for each one.

The important distinction

You are trying to parse:

not:

“known foods”

but:

“measurable entities”

That is the right abstraction.

Best naming choice

I would rename your proposed layer from:
	•	quantified_object_extractor

to either:
	•	quantified_entity_extractor
or
	•	measurable_entity_extractor

I prefer:

quantified_entity_extractor

because it stays broad and composable.

One more design rule to add

Tell Replit this too:

Do not require the entity label to come from a closed dictionary.
The extractor must support open-vocabulary measurable nouns.
Known vocabularies may improve category inference, but they must not be required for extraction.

That line matters a lot. It prevents the system from collapsing back into hand-coded noun lists.

The next best move after this

After quantified entity extraction, the strongest next step is:

entity relation binding

So NOMOS can know not just:
	•	80g cyclic dextrin

but also:
	•	consumed
	•	30 minutes before lifting
	•	as candidate A
	•	under a fast-carb constraint

That is where the quantified entities become part of a real reasoning graph.
