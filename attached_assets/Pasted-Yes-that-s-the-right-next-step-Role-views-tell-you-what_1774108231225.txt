Role views tell you what to look at. Guided workflows tell you what to do next.

That is what turns NOMOS from a dashboard into an operating system.

What to tell Replit

Build guided workflows by role mode for the NOMOS ecosystem cockpit.

Goal:
For each cockpit role mode, provide a recommended operating flow so the user is guided through the most relevant sequence of panels and checks.

This is not automation.
It is deterministic workflow guidance only.

Create:
- src/ui/cockpit/workflow_types.ts
- src/ui/cockpit/workflow_config.ts
- src/ui/components/cockpit/RoleWorkflowPanel.tsx
- update EcosystemCockpitPage.tsx to render the workflow for the active role mode

Requirements:

1. Add canonical workflow types.

type CockpitRoleMode =
  | "builder"
  | "auditor"
  | "governor"
  | "operator";

interface WorkflowStep {
  id: string;
  title: string;
  description: string;
  targetCardId: string;
}

interface RoleWorkflow {
  mode: CockpitRoleMode;
  title: string;
  summary: string;
  steps: WorkflowStep[];
}

2. Define deterministic workflows for each role.

Builder mode:
- inspect attention alert
- open trace / proof source
- inspect invariant break or parser/evaluator mismatch
- inspect formula / health trace if needed

Auditor mode:
- open candidate proof trace
- compare run diff
- inspect health traceability
- verify satisfaction vs verdict consistency

Governor mode:
- review policy recommendation
- inspect doctrine crosswalk
- inspect gains / tradeoffs / risks
- open deliberation summary
- optionally move to governance panel

Operator mode:
- inspect ecosystem health
- inspect current prediction state
- inspect attention alerts
- inspect active policy / governance state if needed

3. Build deterministic function:
- getRoleWorkflow(mode)

4. UI:
Create RoleWorkflowPanel showing:
- role workflow title
- short summary
- ordered steps
- each step links or scrolls to the relevant cockpit card / panel

5. Workflow behavior:
- same underlying data
- same truth
- no mutation
- purely guidance and navigation

6. Keep everything deterministic.
7. Do not use LLM generation.
8. Do not auto-complete workflow steps.
9. This is guidance, not task execution.

Success condition:
Each role mode in NOMOS has a clear recommended flow that helps the user move through the cockpit in a sensible, domain-appropriate order.

Why this matters

This makes each role view usable in practice.

Without workflows:
	•	Builder mode is just a rearranged screen
	•	Governor mode is just highlighted cards

With workflows:
	•	Builder knows where to start when something breaks
	•	Auditor knows how to verify a suspicious result
	•	Governor knows how to move from recommendation to decision
	•	Operator knows where to look first when attention is needed

That is a big improvement.

What the workflows should feel like

Short, explicit, and operational.

Builder
	1.	Inspect current alert
	2.	Open trace/proof source
	3.	Check invariant or parser/evaluator mismatch
	4.	Inspect formula / health trace if needed

Auditor
	1.	Open candidate proof trace
	2.	Compare with prior run
	3.	Inspect health traceability
	4.	Confirm verdict matches evidence

Governor
	1.	Review recommendation
	2.	Review doctrine crosswalk
	3.	Review gains / tradeoffs / risks
	4.	Open deliberation summary
	5.	Proceed to governance action if justified

Operator
	1.	Inspect health
	2.	Inspect current prediction
	3.	Inspect attention alerts
	4.	Review active policy / governance state if needed

That is exactly the right level.

The key principle

Workflows should be:
	•	role-specific
	•	deterministic
	•	navigational
	•	non-authoritative

They should not become:
	•	wizards
	•	hidden automations
	•	silent decision-makers

What comes after this

After guided workflows by mode, the strongest next step is:

session narratives / worklog layer

So NOMOS can record:
	•	which workflow a user followed
	•	which panels they opened
	•	what decision they reached
	•	what rationale they accepted or rejected

That would add a human operational trace on top of the machine trace.
