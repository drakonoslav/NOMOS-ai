Yes — that is the next weak point.

And it is a serious one.

The problem

Right now NOMOS is getting better at:
	•	extracting quantified entities
	•	building a graph
	•	executing graph constraints

But the chain:

quantified entity extraction → tag/class assignment → graph node data → graph filtering

is still fragile.

That means a constraint like:
	•	fast carbs within 90m before lifting >= 60

only works reliably if the system can prove:
	1.	this entity is a carb
	2.	this entity is fast
	3.	that classification survives into the graph
	4.	the graph executor filters on those exact tags

If any link in that chain is implicit, ad hoc, or recreated later, you will get silent semantic drift.

⸻

What the weak point probably is

The likely failure pattern is:

Extraction layer

Entity gets parsed as:
	•	cyclic dextrin
	•	oats

Somewhere later

A heuristic infers:
	•	dextrin = fast carb
	•	oats = slow carb

But:
	•	tags may not be stored canonically on the entity
	•	or not copied into the graph node
	•	or graph filtering may depend on string matching instead of canonical tags
	•	or provenance of that tag may be lost

So the system can “work” in a test case while still being structurally weak.

That is dangerous.

⸻

Why provenance matters

You need to know why something is tagged as:
	•	fast
	•	slow
	•	carb
	•	protein
	•	fluid
	•	supplement
	•	load
	•	duration

Because later, when a graph filter says:
	•	select entities tagged fast and carb

you need to be able to prove:
	•	where those tags came from
	•	whether they were user-declared
	•	registry-derived
	•	inferred
	•	or fallback guesses

Without provenance, the graph executor becomes a black box.

What I suspect is happening now

Probably something like this:
	•	extraction gets cyclic dextrin
	•	some nutrition rule knows dextrin is “fast carb”
	•	graph node maybe stores only label + category
	•	graph constraint executor later filters with something loose like:
	•	label contains “dextrin”
	•	or category == food
	•	or tag recreated during execution

That works in a demo, but it is not stable enough for a real engine.

⸻

What the correct architecture should look like

Extraction output

{
  entityLabel: "cyclic dextrin",
  tags: ["fast", "carb"],
  tagProvenance: {
    fast: "registry",
    carb: "registry"
  }
}

Graph node

{
  type: "entity",
  label: "cyclic dextrin",
  data: {
    tags: ["fast", "carb"],
    tagProvenance: {
      fast: "registry",
      carb: "registry"
    }
  }
}

Graph query

filterEntitiesByTags(["fast", "carb"])

Proof trace

- entity: cyclic dextrin
- matched tags: fast, carb
- provenance: registry, registry

That is the right chain.

⸻

The deeper invariant

You want this law:

classification must happen once, be stored canonically, and be reused downstream

Not:

classify at extraction
classify again at graph build
classify again at execution

That creates drift.
