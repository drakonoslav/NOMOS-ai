Why this matters

This solves a major architectural problem:

before
	•	parser extracts things
	•	graph builder guesses structure
	•	executor queries graph
	•	proof tries to explain graph
	•	UI tries to map back

after
	•	parser creates canonical entities/relations
	•	graph is just a faithful semantic projection
	•	executor queries canonical graph
	•	proof references canonical graph
	•	UI visualizes canonical graph

That is much cleaner.

⸻

The key invariant

You need this law:

The graph may project canonical semantics. It may not invent them.

That is the heart of unification.

If the graph builder is still:
	•	re-tagging
	•	reclassifying
	•	re-binding
	•	guessing candidate structure

then it is still not the semantic spine.

⸻

What this unlocks

Once graph unification is done, a lot of your later systems become much safer:

execution

runs over canonical graph queries

proof

references canonical graph ids and canonical relation types

highlighting

uses graph ids that actually correspond to canonical semantic objects

back-propagation

can show which canonical objects participated in execution

diff engine

can compare graph states canonically

policy / governance / health

all inherit from one coherent substrate

That is the real payoff.

⸻

A subtle design choice

I would keep:
	•	CanonicalEntity
	•	CanonicalRelation

as the actual semantic source of truth

and let:
	•	CanonicalGraph

be the query/visualization substrate

So the graph is central, but not ontologically “more true” than the canonical records it is built from.

That prevents weird feedback loops where the graph becomes harder to reason about than the source.
