==================================================
PRIMARY ARCHITECTURAL RULES
==================================================

1) SINGLE SOURCE OF TRUTH
All evaluative truth must live in:
- packages/constitutional-kernel

That includes:
- parsing outputs after normalization
- candidate evaluation logic
- constraint logic
- scoring logic
- nutrition/domain rule logic
- canonical declaration semantics
- constitutional gating semantics

The dashboard must not carry mirrored business logic that can drift.

2) SOLE EXECUTION GATEWAY
Only this package may execute the kernel:
- artifacts/api-server

The root workspace must not boot the kernel directly.
The dashboard must not directly execute or simulate kernel truth.

3) UI IS PRESENTATION, NOT TRUTH
The dashboard may:
- render results
- format records
- display traces
- provide role-aware views
- collect user inputs
- request parse/evaluate/state endpoints

The dashboard must not:
- reimplement kernel rules
- maintain hand-synced mirrored normalization tables
- maintain duplicate evaluation semantics
- build alternate truth pipelines in parallel to the kernel

4) PERSISTENCE MUST MATCH CLAIMS
Any feature described as:
- immutable audit
- governance history
- worklog persistence
- replay
- traceability
must either:
A) persist durably through a real persistence layer
or
B) be explicitly labeled demo/in-memory/non-durable

Do not leave ambiguous language in docs or code comments.

5) LLM IS ADAPTER, NOT MACHINE TRUTH
If LLM-assisted parsing or proposing exists, treat it as adapter logic only.
That means:
- LLM may extract/suggest/propose
- deterministic kernel types must still normalize and validate final machine state
- final truth must come from typed deterministic kernel pathways
