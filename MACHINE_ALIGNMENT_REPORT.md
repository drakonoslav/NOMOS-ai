## Executive Summary

The **gateway architecture is correct.** All evaluation calls flow:

```
Dashboard → POST /api/nomos/query/evaluate → api-server → nomos-core kernel
```

The dashboard never imports `nomos-core` directly. The API server is the sole execution gateway.

However, the dashboard carries **three categories of secondary logic** that create real drift risk:

1. A constraint algebra implementation in `src/eval/` — not in the UI call path, but tested and maintained in parallel with the kernel's own constraint machinery
2. A protein placement structural audit in `src/evaluation/protein_placement_audit.ts` — claims to be "ground truth check" but runs client-side only
3. A constraint classifier in `src/compiler/constraint_compiler.ts` — a UI-side presentation classifier whose mapping table must stay in sync with the kernel's `constraint_normalizer.ts`

All three are now clearly labeled. No behavior has been removed.

## Part 1 — Duplicated Logic Inventory

### Item 1: Constraint Algebra Engine

| Property | Value |
|----------|-------|
| **Dashboard files** | `src/eval/constraint_algebra.ts`, `src/eval/diff_engine.ts`, `src/eval/baseline_trace.ts`, `src/eval/constraint_report.ts` |
| **Kernel counterpart** | `packages/constitutional-kernel/src/evaluation/constraint_normalizer.ts`, `evaluation/deterministic_matcher.ts`, `evaluation/candidate_scoring.ts` |
| **Classification** | Business-logic duplication |
| **In production UI path** | NO — these files are imported only by their dedicated test files |
| **Drift risk** | HIGH — implements full constraint operators (`MUST_EQUAL`, `SUBSET_OF`, `MINIMIZE_ABS_DELTA`, `SOURCE_PRIORITY`) independently of the kernel |

**Detail:**
The `src/eval/` directory contains a formal constraint algebra with diff functions per variable (protein placement, meal order, meal dispersal, calorie delta). These implement the same conceptual operators as the kernel's `deterministic_matcher.ts`. The code is correct and well-documented. However, it runs only in tests — no UI component currently imports from `src/eval/`. The kernel's own implementation is what runs in production via the API.

**Action taken:** Comment headers updated to state clearly that these files are NOT in the production UI call path. See files for updated headers.

**Recommended future action:** Either:
- Move `src/eval/` into the kernel as a shared contract algebra (preferred — eliminates the parallel implementation)
- Or keep as a test-only reference implementation and remove the claim to be a production ground truth

### Item 2: Protein Placement Structural Audit

| Property | Value |
|----------|-------|
| **Dashboard file** | `src/evaluation/protein_placement_audit.ts` |
| **Kernel counterpart** | `packages/constitutional-kernel/src/nutrition/audit_engine.ts`, `nutrition/macro_engine.ts` |
| **Classification** | Business-logic duplication — secondary validation layer |
| **In production UI path** | YES — used by evaluation_report_builder.ts to enrich display |
| **Drift risk** | HIGH — extracts a "baseline protein placement map" from declared PhasePlan and diffs it against candidate description. If the kernel's nutrition rules change, this layer may disagree. |

**Detail:**
The file describes itself as "the GROUND TRUTH CHECK against which the engine's report is validated." This claim is architecturally incorrect. The kernel's API response IS the ground truth. This module is a secondary UI-side structural check that enriches the display by detecting additional placement detail. It should not be presented as a ground-truth checker — it is a display enrichment layer that may add false positives if its heuristics drift from the kernel's nutrition rules.

**Action taken:** Comment header updated to say "SECONDARY DISPLAY ENRICHMENT LAYER — not a ground truth arbiter."

**Recommended future action:** Expose a per-constraint violation trace from the kernel API response directly, eliminating the need for this client-side structural re-check.

### Item 3: Constraint Classifier (display-side)

| Property | Value |
|----------|-------|
| **Dashboard file** | `src/compiler/constraint_compiler.ts` |
| **Kernel counterpart** | `packages/constitutional-kernel/src/evaluation/constraint_normalizer.ts` |
| **Classification** | Presentation adapter with drift risk |
| **In production UI path** | YES — used to classify constraints for display in the CompiledDraftPanel |
| **Drift risk** | MEDIUM — the mapping table (raw text → constraint kind) must stay in sync with the kernel's normalizer or displayed classification will disagree with evaluation classification |

**Detail:**
The constraint compiler classifies raw constraint strings into display kinds (`STRUCTURAL_LOCK`, `ALLOWED_ACTION`, `TARGET_TOLERANCE`, `SOURCE_TRUTH`, `INTERPRETATION_REQUIRED`) for use in the UI. The kernel's `constraint_normalizer.ts` performs a similar classification to determine which evaluator to invoke. These classifications need to match. If the kernel adds a new constraint kind, the dashboard's classifier must be updated to match.

**Action taken:** Comment header updated to say "UI DISPLAY CLASSIFIER — authoritative classification is in the kernel."

**Recommended future action:** Expose the constraint kind from the kernel API response (`/api/nomos/query/parse`) so the dashboard uses the kernel's classification directly rather than re-classifying client-side.

### Item 4: Eval Types Mirror

| Property | Value |
|----------|-------|
| **Dashboard file** | `src/ui/evaluation/eval_types.ts` |
| **Kernel counterpart** | `packages/constitutional-kernel/src/evaluation/eval_types.ts` |
| **Classification** | Contract/type duplication (NOT business-logic duplication) |
| **In production UI path** | YES — used by QueryBuilderPage, evaluate_nomos.ts, evaluation_mapper.ts |
| **Drift risk** | LOW-MEDIUM — the types define the API response contract shape. If the kernel changes its `EvaluationResult` output shape, this file must be updated. |

**Detail:**
Despite the file comment saying it is a "mirror," the dashboard eval_types and kernel eval_types are actually DIFFERENT shapes. The kernel's `eval_types.ts` defines internal pipeline types (`NormalizedConstraint`, `CandidateEvaluationDraft`, `NormalizedCandidate`). The dashboard's `eval_types.ts` defines the API response format (`CandidateEvaluation`, `EvaluationResult` with `overallStatus`). These are legitimately different: one is internal, one is the external contract.

**Action taken:** Comment header updated to say "API response contract types — these represent what the API returns, not the kernel's internal types."

**Recommended future action:** Generate these types from the OpenAPI spec via `lib/api-spec` → `lib/api-zod`. This would eliminate manual maintenance entirely.

| Property | Value |

**Action taken:** No code change needed. Already labeled correctly in the file header.

## Part 6 — What Is Not Split

These are confirmed clean:

- **Evaluation verdicts** — come from kernel only via API
- **Policy state** — `policy_governance_store.ts` is client-only state management, not a parallel engine
- **Audit records** — `audit_store.ts` stores API results, does not regenerate them
- **Governance audit trail** — records human actions, not evaluation results
- **Health index** — computed in `ecosystem_health_index.ts` from audit records (deterministic formula, no kernel duplication)
- **Worklog/replay** — pure operational trace of UI events, no evaluation semantics
